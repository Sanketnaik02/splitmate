1. PROJECT OVERVIEW
==============================================================================

Application Name : SplitMate
Version          : 1.0.0
Developed By     : Mr. Sanket Naik
Role             : Founder & Developer

Description:
SplitMate is a modern, mobile-first expense sharing and bill splitting
web application. It enables groups of people  friends, roommates, couples,
or colleagues  to track shared expenses, split bills equitably, and settle
debts without the usual friction of manual calculations.

Main Purpose:
To eliminate the awkwardness and complexity of splitting group expenses
by providing a simple, intuitive platform where every member can see who
paid, who owes what, and how to settle up  all in real-time.

Target Users:
- Friend groups on trips or outings
- Roommates managing shared household expenses
- Couples tracking shared finances
- Colleagues splitting team meals or gifts
- Event organizers managing group contributions

Business Use Case:
SplitMate reduces financial friction in group settings. Instead of tracking
expenses in notes or spreadsheets, users create groups, add members, log
expenses, and the app automatically calculates balances, simplifies debts,
and suggests the minimum number of settlements required to zero out everyone.

4. DATABASE STRUCTURE
==============================================================================

SplitMate uses a hybrid storage model:
  - User profiles are stored in Supabase PostgreSQL (server-side)
  - All application data (groups, expenses, members, settlements) is stored
    in browser localStorage using a structured CRUD wrapper

----------------------------------------------------------------------
4.1 Supabase Tables
----------------------------------------------------------------------

Table: profiles

  Column            | Type          | Description
  ------------------+---------------+-------------------------------------------
  id                | UUID (PK)     | References auth.users(id) ON DELETE CASCADE
  email             | TEXT NOT NULL  | User's email address
  display_name      | TEXT          | User's display name
  photo_url         | TEXT          | Avatar/profile photo URL
  phone             | TEXT          | Phone number (default: '')
  default_currency  | TEXT          | Preferred currency (default: 'INR')
  created_at        | TIMESTAMPTZ   | Account creation timestamp

  RLS Policies:
    - SELECT: auth.uid() = id (users can read only their own profile)
    - INSERT: auth.uid() = id (users can create their own profile)
    - UPDATE: auth.uid() = id (users can update their own profile)

  Trigger: handle_new_user()
    - Fires AFTER INSERT ON auth.users
    - Auto-creates a profile row with id, email, and display_name
    - Prevents duplicate profiles with ON CONFLICT (id) DO NOTHING

5. AUTHENTICATION FLOW
==============================================================================

5.1 Email Signup (/signup)

  Step 1: User enters name, email, and password (client-side validation:
          email format check, password minimum 6 characters).
  Step 2: signUp() calls supabase.auth.signUp() with display_name metadata.
  Step 3: If signUp returns a session (auto-confirmed/disable email confirm):
          - Profile insert attempted in profiles table
          - resolveUser() fetches or creates the profile
          - setUser() sets the authenticated user in context
          - Returns user object (truthy) → navigates to /dashboard
  Step 4: If no session returned (email confirmation required):
          - Returns null → shows "Check your email" confirmation screen
  Step 5: Duplicate email → throws "already registered" error →
          getAuthErrorMessage() returns:
          "An account with this email already exists. Please sign in."

5.2 Email Login (/signin)

  Step 1: User enters email and password.
  Step 2: signIn() calls supabase.auth.signInWithPassword().
  Step 3: On success:
          - resolveUser() fetches profile from profiles table
          - setUser() called EXPLICITLY (ensures user is set before
            the function returns, avoiding race conditions with the
            async onAuthStateChange handler)
          - Returns user object → navigate('/dashboard')
  Step 4: Wrong credentials → returns "Invalid email or password."

5.3 Authentication State Persistence (App init)

  Step 1: loading=true, App shows splash/loading state
  Step 2: init() calls supabase.auth.getSession()
  Step 3: If session exists with user:
          - resolveUser() fetches profile
          - setUser() stores user in context
          - safeFinish() sets loading=false
  Step 4: If no session:
          - Does NOT set loading=false immediately
          - Waits for onAuthStateChange to fire
  Step 5: onAuthStateChange handles events:
          - SIGNED_IN / INITIAL_SESSION / TOKEN_REFRESHED:
            resolveUser + setUser + safeFinish()
          - SIGNED_OUT: setUser(null) + safeFinish()
          - INITIAL_SESSION without user: safeFinish()
  Step 6: 10-second safety timeout: forces loading=false if
          auth never resolves (prevents infinite loading)

5.4 Logout

  Step 1: signOut() calls supabase.auth.signOut()
  Step 2: Sets user to null in context
  Step 3: ProtectedRoute detects user=null → redirect to /signin

5.5 Google Login

  Sign-in with Google is implemented via signInWithOAuth({ provider: 'google' }),
  which triggers Supabase's OAuth redirect flow. The redirect URL is set to
  the current origin + /dashboard. Google login is available in the codebase
  but its implementation is on hold pending full email auth production readiness.

5.6 Password Reset

  Forgot Password (/forgot-password):
    - Calls supabase.auth.resetPasswordForEmail()
    - Shows success message with email confirmation instructions
    - Rate-limited with per-email dedup + 3-second cooldown

  Reset Password (/reset-password):
    - Captures new password from user
    - Calls supabase.auth.updateUser() to set new password
    - Cleans up onAuthStateChange listener on unmount

5.7 Rate Limiting

  AuthContext implements a custom rate-limiting system:
  - withDedup(key, fn): Wraps any auth function to prevent concurrent calls
  - pendingRef: Tracks in-flight requests per key
  - lastCallRef: Tracks last call timestamp per key
  - checkCooldown(): Rejects calls within 3 seconds of the last call
  - Applies to: signUp, signIn, signInWithGoogle, signOut

8. CALCULATION ENGINE
==============================================================================

8.1 Split Types

  Equal Split:
    share = totalAmount / numberOfMembers
    Every selected member pays an equal share.

  Percentage Split:
    share = totalAmount * (percentage / 100)
    Each member's percentage must be provided; validated to sum to 100.

  Exact Split:
    share = specifiedAmount
    Each member's exact share is specified directly.

  Shares Split:
    share = totalAmount * (memberShares / totalShares)
    Members are assigned shares (e.g., 2:1:1) and the total is divided
    proportionally.

8.2 Group Balance Calculation

  For each group, balances are computed per member:

    balance[userId] = sum(amount paid by userId)
                    - sum(split share of userId)
                    + sum(settlements from userId)
                    - sum(settlements to userId)

  Where:
    - Amounts paid INCREASE the payer's balance
    - Split shares DECREASE each member's balance (their portion of
      the expense)
    - Settlements from A to B: A's balance increases (they are paying),
      B's balance decreases (they are receiving)

8.3 User Overall Balance (Dashboard)

  Aggregates across ALL groups the user belongs to:

    totalOwed   = sum of positive balances across groups
    totalOwes   = sum of absolute negative balances across groups
    net         = totalOwed - totalOwes
    totalExpenses = sum of all expense amounts in all groups
    totalPaid     = sum of amounts where the user was the payer

8.4 Debt Simplification Algorithm

  Uses a greedy minimum-transactions approach:

  1. Separate members into creditors (positive balance) and
     debtors (negative balance), sorted by amount descending.
  2. Iterate through both lists simultaneously:
     - Take the smallest of creditor[ci].amount and debtor[di].amount
     - Create a transaction from debtor to creditor for that amount
     - Reduce both amounts by that value
     - Advance past members whose balance is now ~zero (< 0.01)
  3. Result is the minimum number of transactions needed to
     zero out all balances.

9. SUBSCRIPTION SYSTEM

9.1 Plans

  Plan        | Price  | Max Groups | Key Features
  ------------+--------+------------+-------------------------------------------
  Free        |  ₹0    |     5      | Basic tracking, equal split, dark mode
  Starter     |  ₹20   |    10      | + percentage/exact split
  Pro         |  ₹100  |    30      | + expense reports, priority support

9.2 Plan Storage

  Plans are stored as constants in config/constants.js.
  User subscription status is persisted in localStorage
  (collection: "subscriptions").

  SubscriptionContext reads the user's subscription from localStorage
  and falls back to 'free' if none found.

9.3 Group Limit Validation

  - groupCount: counted dynamically by filtering groups where the user
    has a membership entry
  - atLimit: true when groupCount >= plan.maxGroups
  - remaining: plan.maxGroups - groupCount

  CreateGroup checks:
    If atLimit → show upgrade modal (redirect to subscription)
    If isLastFreeGroup (Free plan, 4 groups used) →
      show warning modal with upgrade option
    Otherwise → create group normally

10. SECURITY
==============================================================================

10.1 Authentication Security

  - All passwords are handled by Supabase Auth (bcrypt hashing, never
    exposed to the client)
  - Session tokens are managed via Supabase's built-in PKCE flow
  - Anonymous key (VITE_SUPABASE_ANON_KEY) is a publishable key with
    RLS restrictions; it is safe for client-side exposure
  - Rate limiting built into AuthContext (3-second cooldown, dedup)
  - 10-second safety timeout prevents infinite loading if auth fails

10.2 Session Handling

  - Session persistence via Supabase's getSession() on app init
  - Real-time session changes via onAuthStateChange subscription
  - Cleanup: subscription is unsubscribed on unmount
  - INITIAL_SESSION event handled for PKCE completion

10.3 Password Security

  - Minimum 6 characters (client-side validation)
  - Weak password detection via authErrors.js
  - Invalid email format detection

10.4 Database Security (Supabase RLS)

  profiles table:
    - SELECT: authenticated user can only read own row (auth.uid() = id)
    - INSERT: authenticated user can only insert own row
    - UPDATE: authenticated user can only update own row
    - No user can access another user's profile data

10.5 localStorage Security

  - All data is stored in the browser's localStorage with "splitmate_" prefix
  - Data is not encrypted (shared device risk)
  - Arrays are validated before use (Array.isArray check in getAll())
  - Corrupted data does not crash the application (try-catch wrappers)

15. EXECUTIVE SUMMARY
==============================================================================

SplitMate is a production-ready, mobile-first expense sharing platform
built with React 18, Vite, Tailwind CSS, and Supabase Auth. It enables
groups to track shared expenses with support for four split types
(Equal, Percentage, Exact, Shares), automatic balance calculations,
debt simplification, and settlement tracking.

Architecture:
  The application follows a hybrid storage model. User authentication
  and profile management are handled via Supabase (PostgreSQL with
  Row-Level Security), while all application data  groups, members,
  expenses, settlements, and subscriptions  is stored locally in the
  browser's localStorage. This design allows SplitMate to function
  entirely offline without requiring a dedicated backend for core
  operations.

User Experience:
  SplitMate delivers a polished, mobile-optimized experience with
  dark mode support, loading skeletons, toast notifications, empty
  states, and smooth animations. The bottom navigation bar provides
  quick access to the four main sections: Dashboard, Groups, Plans,
  and Profile. A floating action button enables rapid expense logging.

Monetization:
  SplitMate implements a tiered subscription model (Free/Starter/Pro/
  Business) with group-based limits. The Free plan supports up to 5
  groups with basic features. Paid plans start at ₹20/month and unlock
  higher group limits, advanced split types, expense reports, and
  priority support.

Security:
  Authentication is handled entirely by Supabase Auth, which provides
  industry-standard password hashing, PKCE-based session management,
  and OAuth support. The profiles table is protected by Row-Level
  Security policies that ensure users can only access their own data.
  Client-side rate limiting prevents auth abuse.

Status:
  SplitMate v1.0.0 is feature-complete with 42 implemented features
  across 18 pages and 24 components. Google OAuth integration is
  implemented but pending production testing. The application builds
  cleanly (0 errors, 117 modules) and is ready for Vercel deployment.
